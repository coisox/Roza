Layout Terminology

Standard layout terminology for easier communication and bugs reporting.

Creating Module Life Cycle

  1. Prepare UI
  2. Create Landing File
  3. Create Metro Menu
  4. Call Landing File

Prepare UI

UI means what would you like the screens' area to display (usually a listing or form). The area refer to taskbar, leftPanel, rightPanel or fullPanel.

A UI is composed of:

  1. Element - Eg text, dropdown, checkbox, datepicker
  2. Data - What value you want each of the element to be
  3. Lookup - If the element is dropdown, you want to populate all the options

The most basic elements are title, text and textarea. We will use them as our Hello World exercise. Your very first UI will be:

 
[
  {
    "element": "title",
    "labelbm": "Tajuk Saya",
    "labelbi": "My Title"
  },
  {
    "element": "text",
    "labelbm": "Nama Saya",
    "labelbi": "My Name"
  },
  {
    "element": "textarea",
    "labelbm": "Komen Saya",
    "labelbi": "My Comment"
  }
]

This UI must be saved in"dev/ui"folder using".json"extension. Lets say the file name is"ui_1.json". Lets test how it looks in Roza Apps.

In Roza Apps, open browser's console and type this:

 
xxxxxxxxxx
rozaSetPanel({
  panel: 'leftPanel',
  ui: "ui_1.json"
});

Wallaa! You have completed preparing your first UI.

Just now, you UI has been called through browser's console. In real apps, the UI must be called from menu (or user action). So the next step is Create Landing File.

Create Landing File

A landing file is a Javascript file where we define which screens' area our UI should be placed. We could also define any function which will interact with our UI.

All landing file should be saved in"dev/landing"folder using".js"extension. Usually we will call rozaSetPanel() and rozaSetTaskbar() in this file.

rozaSetPanel()

rozaSetPanel() is a build in function to set where our UI should be placed (leftPanel, rightPanel, fullPanel). It takes 3 properties:

 
xxxxxxxxxx
rozaSetPanel({
  panel: 'leftPanel',
  ui: "ui_1.json"
});

rozaSetTaskbar()

rozaSetTaskbar() is used to to defined our taskbar. Only element taskbar can be assigned here. See Element taskbar.

 
xxxxxxxxxx
rozaSetTaskbar({
  ui: "ui_2.json",
  callback: function() {
    console.log('Taskbar is ready');
  }
});

This is a complete sample of landing file"dev/landing/SampleLandingFile.js":

 
​x
rozaSetPanel({
  panel: 'leftPanel',
  ui: "ui_1.json"
});
​
rozaSetTaskbar({
  ui: "ui_2.json"
});

Create Metro Menu

One of the MDEC guideline is maximum 3 level menu. We honor this guideline so the Metro Menu should be defined with at most 2 level menu (we reserve the 3rd level to taskbar).

We must define menu for each module separately (different row in roza_menu table). It is designed such a way so that each developer for each module won't interfere with each other assuming each module is handled by 1 person.

Each 1st and 2nd level menu should has icon as standard layout. The icon use FontAwesome as its class name.

Each menu has access control. See Access Control.

 
xxxxxxxxxx
{
  "icon" : "fa fa-address-book",
  "labelbi" : "Human Resource",
  "labelbm" : "Sumber Manusia",
  "list" : [
    {
      "icon" : "fa fa-user-circle",
      "labelbi" : "Personal",
      "labelbm" : "Peribadi",
      "onclick" : "rozaCallLandingFile('SampleStaffList.js')",
      "ac_remove" : "!rozaHasRole(['admin'])"
    },
    {
      "icon" : "fa fa-sitemap",
      "labelbi" : "B8 Chart",
      "labelbm" : "Carta B8"
    }
  ],
  "ac_remove" : "!rozaHasRole(['admin'])"
}

We will use the Metro Menu to call our landing file.

Call Landing File

rozaCallLandingFile()

Usually we call the landing file from Metro Menu which is defined during Create Metro Menu or from element's event which can be defined during Prepare UI. But you can also call anytime in your Javascript:

 
xxxxxxxxxx
rozaCallLandingFile('SampleStaffInfo.js');

You can also pass parameters as well to be used in the called landing file.

 
xxxxxxxxxx
rozaCallLandingFile('SampleStaffInfo.js?staff_id=13');

Element Type

standardlist

It's an email like listing with 3 visible component; Title, Time, Description. It is usually put at the leftPanel. This element must come with source property point to SQL file saved in "dev/sql" folder.

The standardlist comes with 2 tools (sort and filter). The filter options were defined using "filter" property.

 
xxxxxxxxxx
[
  {
    "element": "standardlist",
    "labelbi": "List of Officer",
    "labelbm": "Senarai Pegawai",
    "onclick": "rozaCallLandingFile('SampleStaff_Form.js?staff_id=[[ROZA_ID]]')",
    "source": "SampleStaff_Standardlist.sql",
    "filter": [
      {
        "element": "text",
        "labelbi": "State",
        "labelbm": "Negeri",
        "parameter": "state"
      },
      {
        "element": "text",
        "labelbi": "Position",
        "labelbm": "Jawatan",
        "parameter": "position"
      },
      {
        "element": "datepicker",
        "labelbi": "Date Appoint",
        "labelbm": "Tarikh Lantikan",
        "parameter": "dateappoint"
      }
    ]
  }
]
​

The source SQL must includes 4 keys (ROZA_ID, ROZA_TITLE, ROZA_DESC, ROZA_TIME) and 1 optional key (ROZA_UNREAD).

Sample of "dev/sql/SampleStandardList.sql":

 
xxxxxxxxxx
SELECT JSON_OBJECT(
    'ROZA_ID', staff_id,
    'ROZA_TITLE', staff_data->>'$.name',
    'ROZA_DESC', CONCAT(staff_data->>'$.position', ', ', staff_data->>'$.group', ', ', (SELECT lov_data->>'$.labelbm' state FROM sample_lov WHERE lov_category = 'state' AND lov_data->>'$.value' = staff_data->>'$.state')),
    'ROZA_TIME', staff_timeupdate,
    'ROZA_UNREAD', 0
)
FROM sample_staff
ORDER BY staff_timeupdate

For prototyping, just hardcode the SQL like this:

 
xxxxxxxxxx
SELECT JSON_OBJECT(
    'ROZA_ID', 1,
    'ROZA_TITLE', 'Sample Tab 1',
    'ROZA_DESC', 'Click me for sample tab',
    'ROZA_TIME', SYSDATE(),
    'ROZA_UNREAD', 0
) FROM DUAL
UNION
SELECT JSON_OBJECT(
    'ROZA_ID', 2,
    'ROZA_TITLE', 'Sample Tab 2',
    'ROZA_DESC', 'Click me for sample tab',
    'ROZA_TIME', SYSDATE(),
    'ROZA_UNREAD', 0
) FROM DUAL
UNION
SELECT JSON_OBJECT(
    'ROZA_ID', 3,
    'ROZA_TITLE', 'Sample Tab 3',
    'ROZA_DESC', 'Click me for sample tab',
    'ROZA_TIME', SYSDATE(),
    'ROZA_UNREAD', 0
) FROM DUAL

ROZA_UNREAD is used when you want to have read (normal font) or unread (bold font) effect. Just like the email client. Set ROZA_UNREAD to 1 for unread, set to 0 or discard the property for read effect.

taskbar

Taskbar is a collection of task grouped into similar behavior. Take a look at these traditional menu:

  1. List of Officer
  2. Officer Registration Form
  3. List of Vendor
  4. Vendor Registration Form

"List of Officer" and "List of Vendor" have similar behavior which is "A Listing". While "Officer Registration Form" and "Vendor Registration Form" have similar behavior which is "A Form". So they can be grouped into:

  1. List (Officer, Vendor)
  2. New Registration (Officer, Vendor)

We can consider element taskbar as our second menu (the first menu is Metro Menu). The way we define is similar to Metro Menu.

    ]

  {

data

By itself, it won't produce any UI at the front end. The element data is used to provide value to all elements defined in the same JSON scope. The data source must return single row non array JSON object.

 
xxxxxxxxxx
[
  {
    "element": "data",
    "source": "SampleStaffBasic.sql"
  }
]

Sample of "dev/sql/DATA_SampleStaffBasic.sql":

 
xxxxxxxxxx
SELECT JSON_MERGE(staff_data, '{"comment":"Testing 123"}')  FROM sample_staff WHERE staff_id = 1

You are advise to use parameterize to avoid SQL injection:

 
xxxxxxxxxx
[
  {
    "element": "data",
    "source": "SampleStaff.sql",
    "parameterize": ["{{staff_id}}"]
  }
]

Sample of "dev/sql/DATA_SampleStaff.sql":

 
xxxxxxxxxx
SELECT JSON_MERGE(staff_data, '{"comment":"Testing 123"}')  FROM sample_staff WHERE staff_id = ?

You can have multiple element data for the same UI. If you are getting data from multiple source, sometimes its more readable to separate them. Keep in mind, all the element data must be put on top of the rest element.

 
xxxxxxxxxx
[
  {
    "element": "data",
    "source": "Source1.sql"
  },
  {
    "element": "data",
    "source": "Source2.sql"
  },
  {
    "element": "text",
    "id": "gender",
    "labelbm": "Jantina",
    "labelbi": "Gender"
  },
  {
    "element": "textarea",
    "id": "comment",
    "labelbm": "Komen",
    "labelbi": "Comment",
    "placeholderbm": "masukkan komen",
    "placeholderbi": "insert comment",
    "maxlength": 255,
    "required": true
  }
]

Sample of "dev/sql/Source1.sql":

 
xxxxxxxxxx
SELECT JSON_OBJECT('gender', 'L') FROM DUAL

Sample of "dev/sql/Source2.sql":

 
xxxxxxxxxx
SELECT JSON_OBJECT('comment', 'Lorem Ipsum') FROM DUAL

html

This special element is used to display custom html markup. It can be anything from normal link to complex graph. The source of it's UI comes from the data element source. If your HTML Markup too complex, put it in external HTML file in "dev > html" folder and then use iframe.

If you want to display a single line of text (eg link) beside it's label, set property style to inline. If you want to display complex html (eg iframe), set property style to block.

 
xxxxxxxxxx
[
  {
    "element": "data",
    "source": {"el1":"<a href='' onclick='download(1)'>My Paperwork.docx</a>"}
  },
  {
    "element": "data",
    "source": {"el2":"<iframe src='dev/html/SampleComplex.html'></iframe>"}
  },
  {
    "id": "el1",
    "element": "html",
    "style": "inline",
    "labelbm": "Pautan",
    "labelbi": "Link"
  },
  {
    "id": "el2",
    "element": "html",
    "style": "block",
    "fullwidth": true
  }
]

This is exampleSampleComplex.html:

 
xxxxxxxxxx
<style> body, html { margin:0; } </style>
<div style="background:red; padding:10px;">Testing 123</div>
<script> alert('Hello!'); </script>

Setting the fullwidth property to true will make the inlinehtml/blockhtml occupy the label area as well.

Please note that some browser will cache iframe content. Meaning that if you change your external html file, you won't see latest changes immediately. Every time you change external html file, it is advisable to add new parameter.

In this example, several iframe calling the same external html file:

 
xxxxxxxxxx
<!-- 1st time call, browser get new content -->
<iframe src="abc.html"></iframe>
​
<!-- the src looks same, browser use cache -->
<iframe src="abc.html"></iframe>
​
<!-- the src looks different, browser get new content -->
<iframe src="abc.html?v1"></iframe>
​
<!-- the src looks same, browser use cache -->
<iframe src="abc.html?v2"></iframe>

If you want every time, the browser get new content from iframe, use {{rozaTimestamp}} as the parameter.

 
xxxxxxxxxx
[
  {
    "element": "html",
    "value": "<iframe src='abc.html?{{rozaTimestamp}}'></iframe>"
  },
]

The {{rozaTimestamp}} will return the current timestamp (epoch) making it unique each time being called. See Predefine Variable.

tabs (depreciated)

Tabs is actually just like menus but layout in such fashion. We define tabs almost exactly like defining taskbar. This element will be automatically set to full width. Any labelbm/labelbi you have specified will be discarded. If you need some kind of label or indicator, use the Element title.

The only labelbm/labelbi you need to specify is on the tabs' items (called list).

Each tab has standard access control, plus with "ac_default" specially used for tabs to set default active tab. "ac_default" accept Javascript condition in string or boolean. See Access Control.

 
xxxxxxxxxx
[
  {
    "element" : "title",
    "labelbi" : "Sample Tab",
    "labelbm" : "Contoh Tab"
  },
  {
    "element" : "tabs",
    "list" : [
      {
        "labelbi" : "Tab 1",
        "labelbm" : "Tab 1",
        "ui" : "UI_Tab1.json",
        "ac_remove" : "!rozaHasRole(['admin'])"
      },
      {
        "ac_remove" : "1==1",
        "labelbi" : "Tab 2",
        "labelbm" : "Tab 2",
        "ui" : "UI_Tab2.json",
        "ac_default": true
      }
    ]
  }
]

accordion

Accordion is actually just like menus but layout in such fashion. We define accordion almost exactly like defining taskbar. This element will be automatically set to full width. Any labelbm/labelbi you have specified will be discarded. If you need some kind of label or indicator, use the Element title.

The only labelbm/labelbi you need to specify is on the accordion' items (called list).

Each accordion has standard Access Control, plus "ac_default" specially used on accordion to set default accordion to be opened. If no "ac_default" being set to any list, the first list will be opened as default.

Element Event

You can use normal jQuery event (eg onclick, onchange, mouseover, etc) in any part of your Javascript. Roza natively provide these events which can be attached to most element.

  1. onclick
  2. onchange
  3. onload
 
xxxxxxxxxx
[
  {
    "id": "state",
    "element": "dropdown",
    "labelbm": "Negeri",
    "labelbi": "State",
    "onchange": "alert(123)"
  }
]

The advantage of using native event compare to normal jQuery is that this event is hidden from the DOM.

Element Callback

If you want to hook your Javascript after a UI has been set, use "onload" property on any element.

Example of"dev/ui/SampleStaff_Form.json":

 
xxxxxxxxxx
[
  {
    "element": "dropdown",
    "id": "state",
    "source": "LOV_SampleState.sql",
    "labelbi": "Element 'dropdown'",
    "labelbm": "Elemen 'dropdown'",
    "onload": "onloadState()"
  },
]

In the above example, the onloadState() function can be defined during creating the landing file.

Example of"dev/landing/SampleStaff_Form.js":

 
xxxxxxxxxx
rozaSetPanel({
    panel: rozaGetParam('staff_id')?'rightPanel':'fullPanel',
    ui: 'SampleStaff_Form.json'
});
​
function onloadState() {
    console.log($('#state').val());
}

Access Control

You can control how an Element or Metro Menu is rendered based on Javascript condition wrapped in string. The condition support Roza's {{param_name}} and [[field_name]].

Depending on element type, not every element has access to every access control.

PropertyTypeDescription
ac_disablestring or booleanIf the condition is met, the element will be disabled (read only)
ac_hidestring or booleanIf the condition is met, the element will be hidden but still exist in DOM
ac_removestring or booleanIf the condition is met, the element will be removed from DOM
ac_defaultstring or booleanSpecially used for tabs > list. f the condition is met, desired tabs > list will be set as active
ac_dualmodestring or booleanIf the condition is met, the element will be editable only in edit mode

Example 1: Disable element if login user id different from passed parameter.

 
xxxxxxxxxx
[
  {
    "element": "text",
    "id": "state",
    "ac_disable": "{{staff_id}}!={{rozaUserId}}"
  }
]

Get Param Value {{param_name}}

During Prepare UI, the JSON stored in database is like a string, thus you cannot call a function. If you try to save this in database, your database client won't accept.

 
xxxxxxxxxx
[
  {
    "element": "dropdown",
    "source": "MySQL.sql",
    "parameterize": [rozaGetParam("myid")]
  }
]

To get parameter in this situation, use double mustache {{param_name}}

 
xxxxxxxxxx
[
  {
    "element": "title",
    "labelbm": "Maklumat Bagi Peribadi {{myid}}"
  },
  {

    "source": "MySQL.sql",
    "parameterize": ["{{myid}}"]
  }
]

You can use {{param_name}} on any properties (eg labelbm, source, onclick, etc).

Get Data Value [[data_name]]

By default, element's value will be assigned based on element data. In this example, the textbox with id = state will get "MEL" as it's default value:

  },
  {
    "element": "text",
    "id": "state"
  }
]

The SQL for "dev/sql/MySQL.sql":

SELECT JSON_OBJECT('state', 'MEL') FROM DUAL

However, the same data value can be used in other properties defined in the same JSON scope:

 
xxxxxxxxxx

rozaSubmitData()

Submit all data in the current panel together with all parameters that has been passed to the landing page to the targeted PHP file stored in "dev/php" folder.

PropertyTypeDescription
filestringFile name where the data will be sent
callback (optional)functionBy default, Roza will popup a modal ("Data has been sent") for successful transaction or a toast for unsuccessful transaction. But you can extend this behavior by using the callback for custom action like custom modal message, clear the form, refresh the standardlist or call another landing file.

This function can be attached to onclick or onchange event during Prepare UI:

 
xxxxxxxxxx
[
  {
    "element": "text",
    "id": "myText",
    "onchange": "rozaSubmitData({\"file\":\"BL_SampleSubmit.php\"}"
  }
]

You can also call through normal jQuery:

 
xxxxxxxxxx
$('#myText').change(function(){
  rozaSubmitData({
    file: "BL_SampleSubmit.php",
    callback: function(data) {
      console.log('Submit data completed');
    }
  });
});

rozaModal()

Modal = Alert = Popup. To open modal, use this in your Javascript:

 
xxxxxxxxxx
rozaModal({
  labelbm: 'Lorem Ipsum',
  labelbi: 'Lorem Ipsum',
  cancel: true, //optional
  onclick: 'okBeingClicked()' //optional
});

labelbm and labelbi can accept HTML markup.

Set cancel to true to display cancel button. Set to false or discard this property to hide cancel button.

If you want OK button that does nothing, set onclick to empty string or simply discard this property.

To close modal, call the same function without any property.

rozalModal();

rozaHasRole()

Check how many user roles (see Predefine Variable: rozaUserRole) intersect with given array. This function usually used in Access Control.

if(rozaHasRole(['admin', 'hr'])) alert('Yea!!!');

Some of the element's properties (eg source, parameterize) will be removed after being process for security measure and smaller data transfer. However, during development, it is advisable for the login user to has "debug" as one of his roles to prevent these remove process thus allowing him to debug in the browser's console.

Use Case

Sharing UI for new and edit form

Usually when an employee fill up leave application form, the same form will be viewed by the HR with minor different (eg the HR will have REJECT button). It's easier to design full UI and set access control based on parameters or roles.

This code shows that, if the landing file received staff_id, it should be displayed on the rightPanel. Otherwise, fullPanel which more suitable for new form.

 
xxxxxxxxxx
rozaSetPanel({
  panel: rozaGetParam('staff_id')?'rightPanel':'fullPanel',
  ui: 2
});

Error Log

All error (front end and back end) will be logged at database (roza_log). A general message will be displayed to the user. For developer, take note on the number in bracket (eg 76). It represent roza_log.log_id which contain details on the error.

File Naming Suggestion

      • HumanResource_Data.js

  2. dev/ui:

    • All UI for any module (eg Human Resource), use:

      • HumanResource_Standardlist.json
      • HumanResource_Taskbar.json
      • HumanResource_Data.json

  3. dev/sql:

    • All lookup SQL (eg State) , use:

      • LOV_State.sql

    • Module SQL (eg Human Resource), use:

      • HumanResource_Standardlist.sql
      • HumanResource_Data.sql
